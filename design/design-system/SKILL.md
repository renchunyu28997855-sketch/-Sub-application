# design-system

## 功能说明

帮助建立完整的设计系统，提供组件规范和设计原则。从设计令牌到原子设计，从组件规范到模式库，构建可扩展、可维护的数字化产品设计体系。

---

## 假设声明（Assumptions）

### 前提条件
- 品牌定位和视觉系统已建立
- 产品类型和平台已明确
- 设计工具已选定（Figma/Sketch等）
- 团队规模和分工已清楚

### 适用范围
- 数字产品设计系统（App/Web）
- 组件库建设
- 设计令牌系统
- 多产品线设计统一
- 设计开发协作规范

### 不适用场景
- 纯品牌视觉系统（用brand-design）
- 纯印刷品设计规范
- 已有完整设计系统的维护

□ 已有设计资产已清点
□ 组件使用情况已统计

## 边界情况（Edge Cases）

### 情况1：多产品线设计系统
**场景：** 需要支持多个独立产品但共享设计系统
**处理：**
```
【多产品线设计系统策略】

产品架构：
- 主产品：[产品A]
- 子产品：[产品B、C]

策略选择：

1. 共享模式
   - 所有产品共享同一设计系统
   - 通过主题/变量区分品牌
   - 适合：品牌统一的产品群

2. 分层模式
   - 共享基础层（Token+原子）
   - 产品特有层（业务组件）
   - 适合：差异较大的产品

3. 联邦模式
   - 各产品独立设计系统
   - 通过规范保持一致性
   - 适合：完全独立的产品

建议：根据产品关联度选择
```

### 情况2：设计系统迁移
**场景：** 已有老系统，需迁移到新系统
**处理：**
```
【设计系统迁移策略】

迁移挑战：
- 历史包袱重
- 团队习惯难改
- 需保持向后兼容

迁移步骤：

1. 摸底阶段
   - 清点现有组件
   - 评估迁移工作量
   - 制定优先级

2. 建设阶段
   - 先建核心组件
   - 新功能用新系统
   - 老功能逐步迁移

3. 替换阶段
   - 分批替换老组件
   - 充分测试验证
   - 预留回滚方案

4. 收尾阶段
   - 清理老组件
   - 更新文档
   - 团队培训

关键原则：渐进式迁移，而非一蹴而就
```

### 情况3：设计开发协作
**场景：** 设计和前端协作不顺畅
**处理：**
```
【设计开发协作规范】

常见问题：
- 设计标注不清楚
- 前端实现有偏差
- 组件状态不完整

解决方案：

1. 交接清单
   - [ ] 设计源文件
   - [ ] 设计标注
   - [ ] 组件规格说明
   - [ ] 交互说明
   - [ ] 异常状态说明

2. 协作工具
   - 设计工具：Figma
   - 组件文档：Storybook
   - 沟通工具：Slack/飞书
   - 任务管理：Jira/Trello

3. 流程规范
   - 设计评审 → 开发 → 验收
   - 设计走查 → 问题反馈 → 修复

4. 设计令牌
   - 打通设计与开发
   - 统一变量命名
   - 自动化转换
```

### 情况4：暗色模式支持
**场景：** 需要同时支持亮色和暗色模式
**处理：**
```
【暗色模式设计系统】

暗色模式设计要点：

1. 令牌层
   - 定义亮/暗两套颜色令牌
   - 语义化命名（background vs background-dark）
   - 自动切换机制

2. 色彩调整原则
   - 降低饱和度
   - 使用深色背景
   - 提高文字对比度
   - 避免纯黑背景

3. 组件适配
   - 所有组件需验证暗色模式
   - 图片/图标需准备深色版本
   - 阴影可能需要调整

4. 实现方式
   - CSS变量
   - Tailwind dark:
   - React Native Dark Mode
```

## 错误处理（Error Handling）

### 错误类型1：系统过于复杂
**识别：** 定义的令牌和组件过多，团队难以使用
**处理：**
```
【系统复杂度评估】

当前问题：系统过于庞大，团队使用成本高

优化方向：

1. 令牌简化
   - 精简颜色数量（≤20个）
   - 精简间距层级（≤6级）
   - 精简圆角层级（≤4级）

2. 组件分级
   - 核心组件：必须使用（5-10个）
   - 扩展组件：推荐使用（20-30个）
   - 高级组件：按需使用（50+个）

3. 分层文档
   - 入门指南（快速上手）
   - 核心规范（必须掌握）
   - 完整文档（按需查阅）

原则：先MVP，后扩展
```

### 错误类型2：组件变体爆炸
**识别：** 组件变体过多（各种组合），难以维护
**处理：**
```
【组件变体管理】

问题：Button组件有10+种变体组合

变体管理策略：

1. 变体分类
   - Type变体：primary/secondary/text
   - Size变体：sm/md/lg
   - State变体：default/hover/active/disabled

2. 组合限制
   - 不是所有组合都有意义
   - 明确声明支持的组合
   - 禁止的组合给出替代方案

3. Props设计
   - 使用Enum定义变体
   - 避免Boolean props爆炸
   - 使用复合Props

4. 代码示例
   enum ButtonVariant { Primary, Secondary, Ghost }
   enum ButtonSize { Small, Medium, Large }
```

### 错误类型3：设计与代码不一致
**识别：** 设计稿和前端实现有差异
**处理：**
```
【设计代码一致性保障】

不一致原因：
- 设计标注不清晰
- 前端实现有误差
- 缺少设计走查

预防措施：
1. 设计阶段
   - 使用组件而非随意画
   - 标注关键尺寸
   - 标注所有状态

2. 开发阶段
   - 使用设计系统组件
   - 参考Storybook实现
   - 遇到疑问及时沟通

3. 验收阶段
   - 设计走查
   - 截图对比
   - 问题记录和修复

工具支持：
- Figma → 代码同步
- Design Token工具
- Chromatic视觉回归测试
```

## FAQ（常见问题）

### Q1：设计系统和组件库有什么区别？
**A：** 组件库是设计系统的核心部分。设计系统更全面，包括：设计原则、令牌系统、组件库、模式库、文档、工具等。组件库侧重于可复用组件，设计系统侧重于完整的设计体系。

### Q2：设计系统应该多大规模？
**A：** 取决于团队和产品：1）MVP阶段：15-20个核心组件；2）发展阶段：30-50个组件；3）成熟阶段：100+组件。原则是"够用就好"，避免过度设计。

### Q3：如何推动团队采用设计系统？
**A：** 推动采用的策略：1）培训：让团队理解系统价值；2）便利：让使用系统比不用更方便；3）激励：把使用系统纳入工作流程；4）迭代：持续优化体验而非强制。

### Q4：设计系统如何保持更新？
**A：** 更新机制：1）设立Owner负责维护；2）制定变更流程；3）版本化管理；4）定期review使用情况；5）收集反馈持续改进。

### Q5：设计系统和品牌系统如何协调？
**A：** 关系：品牌系统定义"品牌层面的视觉规范"，设计系统继承并产品化。设计系统中的颜色、字体、Logo等应来自品牌系统，确保品牌一致性。

### Q6：如何衡量设计系统的价值？
**A：** 衡量指标：1）组件使用率；2）设计效率提升；3）开发效率提升；4）设计一致性提升；5）团队满意度。可通过数据对比和用户调研评估。
