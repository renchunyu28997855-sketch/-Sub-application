# ux-review

## 功能说明

评估产品用户体验，从可用性、可访问性、效率、容错性等维度分析用户流程，识别体验问题并提供可量化的改进建议。

---

## 假设声明（Assumptions）

### 前提条件
- 用户产品已有可体验的原型或上线产品
- 核心用户流程路径已知
- 目标用户群体画像已明确
- 产品类型和平台类型已知

### 适用范围
- 用户注册/登录流程
- 核心功能操作流程
- 电商下单/支付流程
- 内容浏览/搜索流程
- 设置/个人中心流程
- 多步骤向导流程

### 不适用场景
- 仅有静态设计稿无交互原型
- 纯技术架构层面的评估
- 需要A/B测试数据才能判断的决策
- 涉及商业策略的体验设计

## 详细工作流程（Detailed Workflow）

### 阶段1：流程理解与拆解

**步骤1：流程建模**
```
检查点：
□ 确认完整用户路径
□ 识别所有分支路径
□ 标注关键决策点
□ 了解系统反馈机制
```

1. 梳理用户目标
2. 列出完成目标的所有步骤
3. 识别步骤间的分支条件
4. 标注每个步骤的系统反馈

**步骤2：角色分析**
1. 识别流程涉及的用户角色
2. 分析不同角色的权限差异
3. 评估角色切换的体验

**步骤3：环境考虑**
1. 确认使用设备（手机/PC/平板）
2. 考虑使用环境（专注/碎片时间）
3. 评估网络环境依赖

### 阶段2：五维评估

**维度1：可用性（Usability）**

评估标准：
- 用户能否轻松找到需要的功能
- 操作步骤是否符合用户预期
- 界面元素是否易于理解
- 帮助信息是否充分

评估方法：
- 任务完成率
- 错误发生率
- 求助频率

**维度2：可访问性（Accessibility）**

评估标准：
- 符合WCAG 2.1标准
- 色彩对比度达标
- 支持键盘操作
- 支持屏幕阅读器

评估方法：
- 对比度检查
- 焦点顺序检查
- 语义化标签检查

**维度3：效率（Efficiency）**

评估标准：
- 完成任务的步骤数
- 每步操作的耗时
- 完成任务的总时间
- 减少步骤的可能性

评估方法：
- 任务时间测量
- 步骤数统计
- 快捷操作支持

**维度4：容错性（Error Tolerance）**

评估标准：
- 错误预防机制
- 错误提示清晰度
- 错误恢复难易度
- 数据保护机制

评估方法：
- 危险操作确认
- 错误提示有效性
- 撤销/恢复能力

**维度5：满意度（Satisfaction）**

评估标准：
- 视觉美观度
- 动画反馈体验
- 文案友好度
- 整体愉悦度

评估方法：
- 用户反馈分析
- NPS评分预测

### 阶段3：问题分析与输出

**步骤1：问题聚类**
```
检查点：
□ 问题已分类（维度）
□ 严重度已评估
□ 影响范围已确定
```

**步骤2：根因分析**
1. 区分表象和根因
2. 识别设计问题vs实现问题
3. 评估问题优先级

**步骤3：方案生成**
1. 提供可执行的改进建议
2. 说明改进后的预期效果
3. 标注实施复杂度（可选）

## 边界情况（Edge Cases）

### 情况1：缺乏用户流程描述
**场景：** 用户只说"帮我看看这个App的体验"
**处理：**
```
【需要更多信息】

为了进行有效的UX评审，请提供：

1. 核心用户流程
   - 用户想要完成什么目标？
   - 完成目标的主要步骤是什么？

2. 评审重点（可选）
   - [有特定关注的流程请说明]

3. 已知问题（可选）
   - [已发现的问题现象]

如果没有明确的流程，可以选择典型流程进行评审：
- 注册/登录流程
- 核心功能使用流程
- 主要内容消费流程
```

### 情况2：涉及敏感操作流程
**场景：** 评审支付、账号安全等敏感流程
**处理：**
```
【敏感流程评审说明】

[支付/账户安全/隐私]流程涉及敏感操作，需特别关注：

1. 安全与体验的平衡
   - 安全验证的必要性
   - 验证方式的友好度

2. 错误处理
   - 失败原因需要清晰但不泄露安全细节
   - 恢复指导需要具体可操作

3. 资金/数据保护
   - 二次确认机制
   - 撤销窗口期

建议提供：
- 完整的错误提示文本
- 安全验证的触发条件
- 相关的合规要求
```

### 情况3：多平台/多端流程
**场景：** 同一功能在多个平台表现不同
**处理：**
```
【多端流程评审】

您的产品涉及多端（[iOS/Android/Web]），我将按以下方式评审：

1. 各端一致性
   - 核心流程是否一致
   - 功能完整性是否等价
   - 体验水准是否统一

2. 平台特性适配
   - iOS是否遵循Apple HIG
   - Android是否遵循Material Design
   - 平台特有交互是否支持

3. 跨端连续性
   - 多端切换体验
   - 数据同步及时性
   - 状态一致性

建议标注：
- 各端的用户占比
- 重点评审的平台
```

### 情况4：用户流程涉及外部系统
**场景：** 流程中包含第三方支付、地图等服务
**处理：**
```
【涉及外部系统的流程说明】

当前流程涉及第三方服务（[支付/地图/登录等]），评审时：

1. 关注点
   - 跳转到外部系统的时机是否合适
   - 返回后的状态是否清晰
   - 外部服务异常的应对

2. 边界处理
   - 第三方服务不可用时的降级方案
   - 等待第三方响应时的体验
   - 中断后的恢复机制

3. 体验一致性
   - 外部系统与本产品的风格协调
   - 流程衔接的流畅度

我会标注涉及第三方的环节，并给出边界情况建议。
```

## 错误处理（Error Handling）

### 错误类型1：流程描述与实际不符
**识别：** 用户描述的流程与产品实际流程存在差异
**处理：**
```
【流程信息确认】

根据您提供的信息，我梳理的流程如下：

步骤1 → 步骤2 → 步骤3 → ...

请确认：
- 步骤是否正确？
- 是否有遗漏的步骤？
- 分支条件是否准确？

如有差异请指出，我会相应调整评审范围。
```

### 错误类型2：评审范围过于宽泛
**识别：** 要求评审整个App的全部功能
**处理：**
```
【评审范围确认】

UX评审建议聚焦核心用户流程，建议按以下方式选择：

方式A：关键路径评审
- 选择：1-2条核心用户任务流程
- 深度：每个流程6-10步详细分析
- 适合：优化特定转化率

方式B：全流程概览
- 选择：所有主要流程的结构性评审
- 深度：每个流程2-4个关键问题识别
- 适合：全面了解体验现状

建议从方式A开始，聚焦核心问题深入分析。
```

### 错误类型3：缺少基准信息
**识别：** 无法评估"好"还是"不好"
**处理：**
```
【缺少基准信息】

为了给出有价值的改进建议，需要了解：

1. 行业基准（可选）
   - 行业平均转化率是多少？
   - 用户预期完成时间是多少？

2. 历史数据（可选）
   - 改进前的数据如何？
   - 用户反馈主要集中的问题？

3. 竞品对比（可选）
   - 竞品的流程是怎样的？
   - 用户对竞品的评价？

如无上述信息，我将：
- 基于可用性原则进行绝对评估
- 指出违反可用性原则的问题
- 给出行业通用的优化方向
```

## 多个示例（Multiple Examples）

### 典型案例：电商下单流程

**输入：**
```
【UX评审请求】

## 产品信息
- 产品类型：移动电商App
- 产品名称：优购
- 目标用户：20-35岁女性消费者

## 评审范围
- 评审流程：商品下单流程
- 核心路径：浏览商品 → 加入购物车 → 结算 → 支付

## 已知信息
- 当前转化率约35%
- 希望提升到40%以上
```

**输出：**
```
【UX评估报告：电商下单流程】

## 流程概述
- 流程名称：商品下单流程
- 步骤数量：8步
- 预计耗时：2-4分钟
- 流程图：
  [浏览商品] → [加入购物车] → [去结算] → [确认订单] → [选择地址] → [选择支付] → [确认支付] → [下单成功]

## 评估维度

### 1. 可用性（★★★☆☆）
- 发现问题：
  - 地址选择需跳转新页面，打断流程
  - 支付方式选项说明不够清晰
- 改进建议：
  - 地址选择改为半屏弹窗选择
  - 支付方式增加图标和简短说明

### 2. 可访问性（★★★★☆）
- 发现问题：
  - 部分按钮文字与背景对比度略低
- 改进建议：
  - 确保所有文字符合WCAG AA标准（4.5:1）

### 3. 效率（★★★☆☆）
- 发现问题：
  - 确认订单页需下拉才能看到全部信息
  - 优惠券需跳转到新页面选择
- 改进建议：
  - 关键信息前置，减少滚动
  - 优惠券改为弹窗内选择

### 4. 容错性（★★★☆☆）
- 发现问题：
  - 库存不足时无明确提示
  - 支付失败后恢复路径不清晰
- 改进建议：
  - 实时显示库存状态
  - 支付失败页提供明确重试入口

### 5. 满意度（★★★★☆）
- 发现问题：
  - 下单成功页较为平淡
- 改进建议：
  - 增加成功状态的视觉反馈
  - 提供订单追踪入口

## 问题详解

### 问题1：地址选择打断流程
- 严重度：高
- 现象描述：选择收货地址需跳转新页面，返回后需重新确认之前页面信息
- 用户影响：增加用户认知负担，可能导致放弃下单
- 根本原因：地址选择作为独立页面而非流程内组件
- 解决方案：改为底部弹窗选择，选择后直接回填

### 问题2：优惠券选择效率低
- 严重度：中
- 现象描述：选择优惠券需跳转新页面，页面加载后才显示可用优惠券
- 用户影响：增加步骤，降低优惠感知
- 根本原因：优惠券数据未预加载，UI交互过度设计
- 解决方案：弹窗预加载优惠券，支持直接在弹窗内选择和查看优惠金额

### 问题3：库存状态不透明
- 严重度：高
- 现象描述：用户下单时才能发现库存不足
- 用户影响：期望落空，可能导致用户流失
- 根本原因：库存状态未实时同步显示
- 解决方案：在商品列表和详情页显示库存状态（紧张/充足）

## 优化建议

### 流程级优化
1. 将地址选择、优惠券选择改为弹窗式交互
2. 关键信息（价格、优惠、库存）始终可见
3. 增加进度指示器，让用户知道流程位置

### 步骤级优化
| 步骤 | 问题 | 优化方案 | 优先级 |
|------|------|----------|--------|
| 加入购物车 | 无加入成功反馈 | Toast提示+购物车数量更新 | P1 |
| 确认订单 | 信息需滚动查看 | 关键信息前置，折叠详情 | P0 |
| 选择地址 | 跳转新页面 | 改为弹窗选择 | P0 |
| 选择优惠券 | 跳转新页面 | 改为弹窗选择 | P1 |
| 确认支付 | 无倒计时提示 | 增加支付限时提示 | P2 |
| 下单成功 | 页面平淡 | 增加成功动效和快捷操作 | P1 |

### 预期收益
- 完成率提升：预计从35%提升至40-42%
- 操作时间减少：预计减少15-20秒
- 页面放弃率降低：预计降低8-10%
```

### 边界案例：分步注册流程

**输入：**
```
帮我看看这个App的注册流程体验
流程：手机号 → 验证码 → 设置密码 → 基本信息 → 完成
目标用户：中年商务人士
```

**输出：**
```
【UX评估报告：注册流程】

## 流程概述
- 流程名称：用户注册流程
- 步骤数量：5步
- 目标用户：中年商务人士
- 流程图：
  [手机号] → [验证码] → [设置密码] → [基本信息] → [完成]

## 评估维度

### 1. 可用性（★★★★☆）
- 发现问题：
  - 验证码输入框未自动聚焦
  - 倒计时显示不够醒目
- 改进建议：
  - 每个步骤自动聚焦对应输入框
  - 倒计时使用更大字号和对比色

### 2. 可访问性（★★★★★）
- 优点：
  - 文字大小适合中年用户
  - 颜色对比度良好
- 建议：
  - 保持现有标准

### 3. 效率（★★★☆☆）
- 发现问题：
  - 步骤5"基本信息"字段较多，页面较长
  - 部分信息（如职业）用户不确定是否必填
- 改进建议：
  - 精简必填字段
  - 必填项明确标注*

### 4. 容错性（★★★★☆）
- 发现问题：
  - 密码强度提示不够直观
  - 验证码输入错误无即时提示
- 改进建议：
  - 实时显示密码强度（弱/中/强）
  - 验证码4个格子逐一验证

### 5. 满意度（★★★★☆）
- 优点：
  - 每步有进度指示
  - 步骤标题清晰
- 建议：
  - 完成页增加鼓励性文案

## 问题详解

### 问题1：基本信息页字段过多
- 严重度：高
- 现象描述：页面包含10个输入字段，用户感到压力
- 用户影响：中年用户可能因此放弃注册
- 根本原因：收集信息过于贪全
- 解决方案：区分必填和选填，必填仅保留4-5个核心字段

### 问题2：进度指示不够清晰
- 严重度：中
- 现象描述：用户不知道当前在第几步
- 用户影响：不确定还要填多久
- 根本原因：进度指示仅显示点，未显示步骤名称
- 解决方案：进度条显示步骤名称，如"1/5 手机验证"

## 优化建议

### 流程级优化
1. 精简基本信息页至4个必填字段
2. 其他信息延后到个人中心补充
3. 增加每步的预计时间提示

### 步骤级优化
| 步骤 | 问题 | 优化方案 | 优先级 |
|------|------|----------|--------|
| 手机号 | 输入格式无提示 | 实时显示手机号格式是否正确 | P1 |
| 验证码 | 无自动聚焦 | 自动弹出数字键盘 | P0 |
| 验证码 | 倒计时不醒目 | 增大字号，使用品牌主色 | P1 |
| 密码 | 强度不直观 | 添加实时强度指示器 | P0 |

| 基本信息 | 必填不明确 | 添加*必填标识 | P0 |
| 完成 | 无后续引导 | 提供"去使用"按钮 | P1 |
```

## FAQ（常见问题）

### Q1：UX评审和可用性测试有什么区别？
**A：** UX评审是专家评估方法，基于可用性原则和设计经验进行分析。可用性测试是用户研究方法，通过观察真实用户使用产品来发现问题。简单说，UX评审是"专家看病"，可用性测试是"用户看病"。两者互补，建议结合使用。

### Q2：没有原型可以评审吗？
**A：** 可以进行理论层面的评审，但效果有限。UX评审的核心是分析用户与界面的交互，没有原型时，可以基于流程描述和业界最佳实践给出建议。但具体交互问题（如按钮位置、手势设计）需要原型才能准确评估。

### Q3：如何评估抽象的功能（如品牌感知）？
**A：** 品牌感知属于高阶体验要素，需要通过多个触点综合评估。UX评审会从视觉一致性、文案调性、交互反馈等方面分析品牌体验，并指出可能影响品牌感知的具体设计问题。

### Q4：如何平衡用户体验和商业目标？
**A：** 两者本质不冲突，好的用户体验会促进商业目标达成。评审时会指出可能影响转化或商业目标的体验问题，并分析用户体验优化如何支撑商业目标。如果确实存在冲突（如安全验证与便捷性的平衡），会提供多种方案并说明各方案的影响。

### Q5：UX评审需要多久出结果？
**A：** 取决于评审范围的复杂度。单一核心流程的评审通常在5-10分钟内完成。涉及多流程或整个产品时，可能需要15-20分钟。建议聚焦核心流程进行深度评审。

### Q6：如何验证评审建议的有效性？
**A：** 建议采用小范围测试验证：1）在原型上实施优化；2）邀请3-5名目标用户进行快速测试；3）收集反馈并迭代；4）确认有效后再全面推广。无法进行用户测试时，可参考预期收益中的行业基准数据进行预估。

## 与其他技能的关联（Cross-skill References）

### 1. ui-feedback（UI反馈）
**关系：** 视觉层面的评审
**使用场景：** UX评审发现视觉层面的问题后
**推荐顺序：** ux-review发现体验问题 → ui-feedback评审视觉实现

### 2. design-system（设计系统）
**关系：** 系统化解决方案
**使用场景：** 发现大量一致性问题，需要建立设计规范
**推荐顺序：** 多个流程的ux-review → design-system建立规范

### 3. slide-deck（演示文稿）
**关系：** 评审结果输出
**使用场景：** 需要向团队汇报UX评审结果
**推荐组合：** 完成ux-review → 使用slide-deck制作汇报PPT

### 4. user-research（用户研究）
**关系：** 前置研究支持
**使用场景：** 需要真实用户数据支持评审结论
**推荐顺序：** user-research收集数据 → ux-review分析 → ui-feedback优化

### 5. information-architecture（信息架构）
**关系：** 结构层面支持
**使用场景：** UX问题源于信息架构不合理
**推荐组合：** ux-review发现架构问题 → 建议参考IA原则调整
