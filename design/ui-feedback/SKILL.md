# ui-feedback

## 功能说明

对UI设计提供专业评审和改进建议，覆盖布局、配色、字体、交互等维度，帮助识别设计问题并提供可执行的优化方案。

---

## 假设声明（Assumptions）

### 前提条件
- 用户已有具体的设计稿（图片、Figma链接、描述等）
- 目标平台和使用场景已明确
- 目标用户群体已定义

### 适用范围
- 移动App界面（iOS/Android）
- Web页面和后台系统
- 小程序和H5页面
- 部分桌面应用界面

### 不适用场景
- 纯概念性、缺乏具体设计稿的评审
- 需技术可行性验证的复杂交互动画
- 品牌战略层面的设计决策

## 详细工作流程（Detailed Workflow）

### 阶段1：信息收集与确认
```
检查点：
□ 设计稿信息完整（平台、页面类型）
□ 目标用户已明确
□ 核心功能路径已知
□ 已有设计规范的需提供
```

**步骤：**
1. 解析设计稿内容，识别页面结构和元素
2. 确认设计平台和目标设备
3. 了解产品背景和使用场景
4. 如信息不足，询问补充

### 阶段2：系统性评审（按维度）

**步骤1：布局结构评审**
- 检查视觉重量平衡
- 评估信息密度和留白
- 验证对齐和间距一致性
- 检查响应式布局合理性

**步骤2：视觉层次评审**
- 评估标题、正文、辅助文字的层级关系
- 检查内容主次是否清晰
- 验证视觉动线合理性
- 检查重点元素是否突出

**步骤3：色彩评审**
- 验证色彩系统一致性
- 检查对比度是否达标（WCAG 2.1）
- 评估色彩情感是否符合产品调性
- 检查色盲友好性

**步骤4：字体评审**
- 验证字体层级是否清晰
- 检查字号是否合适（最小14px）
- 评估行高和字间距
- 检查中英文混排协调性

**步骤5：交互体验评审**
- 评估按钮和可点击区域尺寸
- 检查反馈机制是否完善
- 验证状态变化是否清晰
- 评估操作路径是否合理

**步骤6：一致性评审**
- 检查设计元素是否统一
- 验证相同功能的一致性
- 检查跨页面导航体验

### 阶段3：问题聚合与输出
```
检查点：
□ 所有维度已评审
□ 问题严重度已评估
□ 改进建议已提供
□ 优先级已排序
```

## 边界情况（Edge Cases）

### 情况1：输入为空
**场景：** 用户只说"帮我看看这个设计"
**处理：**
```
【需要更多信息】

为了提供准确的UI反馈，请提供以下信息：

1. 设计稿
   - 图片文件（Figma截图、设计稿图片）
   - 或 Figma 链接

2. 设计背景
   - 平台类型：[移动App/Web/桌面/小程序]
   - 页面类型：[首页/详情页/登录页等]
   - 目标用户：[描述]

3. 评审重点（可选）
   - [有特定关注点请说明]

感谢配合！
```

### 情况2：输入格式错误
**场景：** 提供了非设计相关的输入（如纯文本需求文档）
**处理：**
```
【输入类型不匹配】

您提供的内容看起来是[需求文档/开发代码/其他]，而非UI设计稿。

UI反馈技能需要具体的设计稿作为评审对象：
- 支持：设计图片、Figma链接、页面截图
- 不支持：纯文本描述、代码实现

请提供可评审的设计稿，我会立即开始评审。
```

### 情况3：特殊设计风格
**场景：** 用户要求评审极其抽象或艺术化的设计
**处理：**
```
【设计风格说明】

当前设计采用[抽象/艺术化/实验性]风格，这可能与常规可用性标准有所不同。

建议评审维度：
1. 艺术表达力：是否符合设计意图
2. 功能可辨识：关键功能是否可识别
3. 用户预期：是否符合目标用户审美
4. 场景适配：在实际使用环境中效果如何

如需调整，我将按上述维度进行评审。
```

### 情况4：复杂交互动画
**场景：** 需要评审的页面包含复杂交互动画
**处理：**
```
【交互动画评审说明】

复杂交互动画的评审需要考虑：
1. 动画目的性：是否有明确的交互目的
2. 性能影响：动画是否影响页面性能
3. 用户理解：用户是否能理解动画含义
4. 可访问性：是否提供不依赖动画的备选

建议提供：
- 动画原型视频或GIF
- 动画触发的具体场景描述
- 目标用户的动画偏好背景
```

## 错误处理（Error Handling）

### 错误类型1：设计稿无法加载
**识别：** 提供的图片链接失效或文件损坏
**处理：**
```
【设计稿加载失败】

无法加载设计稿，请尝试：
1. 重新上传图片（支持PNG/JPG/WebP）
2. 检查Figma链接是否设置了访问权限
3. 提供其他可访问的设计稿来源

临时替代方案：
- 详细描述页面结构和设计元素
- 标注关键设计细节
```

### 错误类型2：平台类型不明确
**识别：** 无法确定目标平台（iOS/Android/Web）
**处理：**
```
【平台类型确认】

评审标准需要适配具体平台，请确认：

□ iOS App（遵循Apple HIG）
□ Android App（遵循Material Design）
□ Web页面（响应式设计）
□ 跨平台设计（多端适配）

不同平台的评审标准有所差异，明确平台后我可提供更精准的反馈。
```

### 错误类型3：评审范围不清晰
**识别：** 涉及多个页面或整体设计系统
**处理：**
```
【评审范围确认】

您涉及多个页面/整体设计系统，建议分批评审：

方案A：整体评审
- 关注点：设计一致性、整体体验
- 适合：早期设计方向确认

方案B：重点页面评审
- 关注点：关键页面深度分析
- 适合：核心用户流程优化

方案C：组件级评审
- 关注点：可复用组件规范性
- 适合：设计系统搭建

请选择评审方案，或指定最重要的页面/组件。
```

**输出：**
```

- 综合评分：★★☆☆☆
- 核心问题：信息过载、层次不清、CTA不够突出
- 建议优先级：P0

## 分项评审

### 1. 布局结构（★★☆☆☆）
- 问题：首屏信息密度过高，包含8个功能入口、3个运营位、1个广告
- 建议：精简首屏内容，控制在5个以内核心功能入口

### 2. 视觉层次（★★☆☆☆）
- 问题：标题(H1)和正文(P)字号差异小，对比度不足
- 建议：H1使用24-28px，Body使用14-16px，增强对比

### 3. 色彩（★★★☆☆）
- 问题：品牌主色不明确，使用了5种以上颜色
- 建议：确定1个主色，辅助色不超过2种

### 4. 字体（★★★☆☆）
- 问题：正文字号偏小（12px），阅读吃力
- 建议：正文字号调整为14-16px

### 5. 交互（★★★☆☆）
- 问题：底部TabBar图标尺寸偏小（20px）
- 建议：图标尺寸统一为24x24px

### 6. 一致性（★★★★☆）
- 优点：卡片圆角统一为12px，间距一致性好

## 问题详解

### 问题1：首屏信息过载
- 现象：用户打开App后需要处理大量信息
- 影响：用户决策成本高，可能导致流失
- 建议：按用户重要性排序，折叠次要功能

### 问题2：核心CTA不突出
- 现象："立即购买"按钮视觉权重与普通按钮相近
- 影响：转化率可能偏低
- 建议：增加按钮尺寸（高度≥48px），使用主色填充

## 修改优先级
| 优先级 | 问题 | 预计影响 |
|--------|------|----------|
| P0 | 首屏信息精简 | 留存率+10% |
| P0 | CTA按钮强化 | 转化率+5% |
| P1 | 字体层级调整 | 阅读体验提升 |
| P2 | 色彩系统精简 | 品牌认知度+ |

## 优化后的预期效果
预计用户首屏停留时间降低（更高效），核心转化率提升8-15%
```

## FAQ（常见问题）

### Q1：UI反馈和UX评审有什么区别？
**A：** UI反馈侧重于视觉层面（布局、色彩、字体、图标等），是设计稿层面的评审。UX评审侧重于用户体验和交互流程。UI是视觉表现，UX是使用体验。两者可以结合使用，建议先用ui-feedback评审视觉，再用ux-review评审流程。

### Q2：可以评审Figma链接吗？
**A：** 可以。请确保Figma链接设置了"任何人都可查看"的权限。如果链接需要登录才能访问，请提供截图作为备选。

### Q3：如何提高评审的准确性？
**A：** 提供以下信息可提高评审质量：
- 明确的目标平台和设备
- 目标用户画像
- 产品设计规范（如有）
- 重点评审维度
- 产品背景和使用场景

### Q4：评审结果是否考虑开发实现难度？
**A：** 评审结果主要从设计角度出发，会标注可能影响开发实现的因素（如复杂动画、特殊字体等），但不深入评估技术可行性。如需技术评审，请结合技术团队意见。

### Q5：可以评审修改后的设计吗？
**A：** 可以。请提供修改前后的对比，标注具体改动点，我可以针对改动进行增量评审，高效验证修改效果。

### Q6：如何处理主观审美差异？
**A：** 评审基于设计原则和用户研究，不带入个人审美偏好。如存在"设计风格选择"类问题，会提供多个方向的利弊分析，由您最终决策。
