# data-analyst

## 功能说明
帮助分析数据，发现趋势、模式和洞察，支持业务决策。

---

## 假设声明（Assumptions）

1. **数据可获取性**：用户已拥有或能够获取待分析的数据集
2. **数据基本质量**：数据包含可分析的信号（非完全随机噪声）
3. **业务背景理解**：用户对分析目标的业务背景有基本了解
4. **工具环境**：分析可在Python/R/SQL/Excel等常见工具中完成
5. **时间投入**：复杂分析需要足够的时间进行探索和验证
6. **数据量假设**：适用于从几十行到数亿行的数据集（规模不同时方法论不同）

---

## 输入输出（I/O Format）

### 标准输入格式

```
【分析目标】具体的业务问题或分析目的
【数据来源】数据表名/文件路径/数据库连接
【数据时间范围】起始日期 ~ 结束日期
【关键指标】需要关注的核心指标列表
【分析维度】待分析的维度（如：地区、渠道、用户类型）
【输出要求】报告形式/可视化类型/交付期限
```

### 标准输出格式

```
【分析目标】XXX
【数据概况】数据量、时间范围、字段数量
【分析方法】使用的分析技术和工具
【核心发现】按重要性排序的关键洞察（3-5条）
【数据可视化建议】推荐的图表类型和建议展示方式
【业务建议】基于数据的具体行动建议
【风险提示】分析结论的局限性或不确定性
【后续建议】可进一步探索的方向
```

## 详细工作流程（Detailed Workflow）

### 第一阶段：问题定义
1. **明确分析目标**
   - 与业务方确认核心问题
   - 确定分析的业务背景和决策场景
   - 定义成功标准

2. **确定分析边界**
   - 确认分析的时间范围
   - 确定分析的对象群体
   - 界定分析的空间范围（部门/产品线/渠道）

3. **输出**：分析需求文档（1页）

### 第二阶段：数据准备
4. **数据获取**
   - 从数据库/文件/API提取数据
   - 记录数据来源和提取时间
   - 保存原始数据备份

5. **数据质量检查**
   - 检查数据量级是否合理
   - 验证关键字段的完整性
   - 识别明显的数据异常

6. **数据清洗**（参见 data-cleaner 技能）
   - 处理缺失值
   - 去除重复记录
   - 统一数据格式

**检查点**：确认数据已准备好进入分析阶段

### 第三阶段：探索性分析（EDA）
7. **描述性统计**
   - 计算各指标的均值、中位数、众数
   - 分析数据的分布特征
   - 识别数据的偏态和峰度

8. **趋势分析**
   - 时间序列趋势（日/周/月趋势）
   - 同比、环比变化
   - 季节性模式识别

9. **对比分析**
   - 按维度分组对比
   - 寻找差异和异常
   - 相关性分析

### 第四阶段：深度分析
10. **模式识别**
    - 聚类分析（参见 stats-model 技能）
    - 分类规则发现
    - 关联规则挖掘

11. **因果推断**
    - 归因分析
    - 假设检验
    - A/B测试结果解读（参见 ab-test 技能）

12. **预测分析**（可选）
    - 趋势外推
    - 回归预测
    - 机器学习模型（参见 stats-model 技能）

### 第五阶段：洞察提炼
13. **发现排序**
    - 按重要性和可信度排序
    - 区分统计显著和业务显著
    - 排除干扰因素

14. **建议生成**
    - 每个发现对应的业务建议
    - 建议的优先级排序
    - 预期效果量化

### 第六阶段：报告输出
15. **报告撰写**
    - 使用标准报告模板
    - 图表与文字配合
    - 控制信息密度

16. **评审与修订**
    - 与业务方确认结论
    - 补充潜在风险
    - 完善建议可行性

## 边界情况（Edge Cases）

### 输入为空的情况
- **无数据**：返回"当前时间范围内无数据，建议扩大时间范围或检查数据源"
- **数据量过小**（<30条）：标注统计显著性不足，仅做描述性分析
- **数据时间跨度不足**：说明季节性分析受限，仅供参考

### 输入格式错误的情况
- **目标不明确**：主动提问澄清，例如"您希望分析哪个业务指标的变化趋势？"
- **时间范围错误**（结束日期早于开始日期）：自动交换或提示修正
- **字段名称不符**：列出可用字段供用户选择

### 特殊情况处理
- **数据存在明显gap（某段时间无数据）**：标注并单独说明，可能影响趋势判断
- **多个数据源口径不一致**：统一数据定义，明确以哪个为准
- **数据涉及敏感信息**：进行脱敏处理后再分析

---

## 错误处理（Error Handling）

| 错误类型 | 可能原因 | 处理方式 |
|---------|---------|---------|
| SQL执行失败 | 表名/字段名错误、权限不足 | 检查SQL语法，确认数据源可用性 |
| 数据量超预期 | 未设置LIMIT、未预估数据量 | 分批处理，设置采样策略 |
| 数值溢出 | 计算结果超出数据类型范围 | 使用高精度类型，拆分计算 |
| 图表生成失败 | 数据格式不支持、内存不足 | 简化图表类型，尝试其他工具 |
| 分析结论矛盾 | 数据质量问题、分析方法不当 | 复核数据质量，审视分析假设 |

## 多个示例（Multiple Examples）

### 典型案例

**场景**：电商平台用户留存分析

**输入**：
```
【分析目标】分析2024年新用户的留存情况，找出流失原因，提出提升建议
【数据来源】dwd_userAction, dwd_userProfile
【数据时间范围】2024-01-01 ~ 2024-03-31
【关键指标】次日留存率、7日留存率、30日留存率
【分析维度】用户来源渠道、用户画像（年龄/性别/地区）
```

**输出**：
```
【分析目标】2024年新用户留存分析与提升建议
【数据概况】新增用户50万，行为记录5000万条，用户画像50字段

【核心发现】
1. 整体留存偏低：次日留存38%（行业均值45%），30日留存8%
2. 渠道差异显著：自然流量用户留存比付费渠道高30%
3. 关键行为节点：完成首次购买的用户30日留存率达55%，未购买用户仅3%
4. 流失高发期：注册后3天内未完成关键行为的用户流失率达70%

【业务建议】
1. 优先级P0：优化新用户首次购买体验，目标将首次购买率从22%提升至40%
2. 优先级P1：调整付费渠道获客策略，提高自然流量占比
3. 优先级P2：建立流失预警机制，对高流失风险用户触发挽留流程

【预期效果】
按P0方案实施后，预计30日留存率可提升至15%，年化增收约3000万
```

### 边界/异常案例

**场景**：数据存在明显异常值

**输入**：
```
【分析目标】分析用户月均消费分布
【数据概况】1万用户，字段：user_id, monthly_amount
```

**发现**：数据中存在3个用户月均消费>100万，明显异常

**处理方式**：
```
【处理说明】
1. 识别异常值：使用IQR方法识别出3个极端高值
2. 分析异常原因：确认为企业账户，与普通用户不同质
3. 处理决策：将分析范围限定为月消费<10万的个人用户
4. 单独说明：异常值单独统计，不影响整体分布结论
```

### 失败案例及修正

**场景**：初次分析结论被推翻

**问题**：分析认为"周末运营活动无效"，但被业务方质疑

**复盘发现**：
1. 未控制其他变量（如：周末天气、节假日因素）
2. 对比维度不一致（周末活动VS工作日活动，用户群体不同）
3. 数据时间跨度不足（仅分析4周数据）

**修正后**：
```
【修正说明】
1. 增加数据时间跨度至12周
2. 使用同期对比（同是周末，有活动vs无活动）
3. 加入天气、节假日控制变量
4. 结论修正：排除干扰因素后，周末活动实际提升转化率12%
```

## FAQ（常见问题）

**Q1：数据分析和BI报表有什么区别？**
A：数据分析侧重于深度分析问题根因，探索未知规律；BI报表侧重于监控已知指标，实时展示现状。数据分析产出洞察和建议，BI报表产出标准化监控。

**Q2：分析结果与业务直觉不符怎么办？**
A：首先复核数据质量和分析方法；然后深入分析是否遗漏了重要的控制变量；如果数据确实支持结论，可以提出"数据发现与直觉不符，需要进一步验证假设"。

**Q3：分析时间太短不够深入怎么办？**
A：采用"快速+深度"策略：先产出快速结论满足紧急需求，再安排时间做深度分析；在报告中明确说明"本次为初步结论，后续将深入验证"。

**Q4：如何判断分析结论是否可信？**
A：检查三点：(1)数据样本量是否足够 (2)分析方法是否适合 (3)结论是否经过交叉验证。可信结论应该能够解释大部分数据变异，且在其他时间段/群体上也能复现。
