# okr-helper

## 功能说明
帮助制定OKR（Objectives and Key Results），追踪进度并协助复盘。

---

## 1. 假设声明（Assumptions）

### 前提条件
- 用户有明确的业务背景和团队情况
- 目标周期明确（季度/半年/年）
- 团队成员理解OKR理念
- 有一定的数据追踪能力

### 适用范围
- 公司级OKR
- 部门级OKR
- 团队级OKR
- 个人级OKR
- 跨部门协作OKR

### 不适用场景
- 不理解OKR理念的团队（需要先培训）
- 只需要KPI的绩效管理
- 纯执行没有自主决策的岗位
- 需要详细任务分解的项目管理

### OKR理念假设
- OKR不是绩效考核
- OKR是目标管理工具
- OKR鼓励挑战性目标
- OKR需要公开透明

## 2. 输入输出（I/O Format）

### 输入格式

**必需信息**：
```
组织层级：[公司/部门/团队/个人]
目标周期：[Q1/Q2/Q3/Q4/半年/年]
制定者：[姓名/角色]
```

**可选补充**：
- 上周期OKR（延续性目标）
- 业务背景和优先级
- 团队规模和能力
- 已有数据指标

### 输出格式

```
【周期】2024年Q1
【层级】公司/部门/个人
【OKR正文】
【执行建议】
【追踪机制】
...
```

## 3. 详细工作流程（Detailed Workflow）

### 阶段一：准备（第1-2步）

```
步骤1：了解业务背景
├── 上周期OKR回顾
├── 当前业务优先级
├── 团队能力和资源
├── 关键挑战和机会
└── 检查点：是否理解业务上下文

步骤2：确定目标方向
├── 明确3-5个核心目标方向
├── 对齐公司/上级OKR
├── 考虑团队职责和影响力
└── 检查点：目标方向是否清晰
```

### 阶段二：OKR制定（第3-5步）

```
步骤3：制定O（Objective）
├── 每个O应该是鼓舞人心的
├── 语言简洁，定性描述
├── 对齐公司和上级目标
├── 每个周期3-5个O
└── 检查点：O是否清晰有激励性

步骤4：拆解KR（Key Results）
├── 每个O对应2-4个KR
├── KR必须是可量化的
├── KR应该有挑战性（完成70%是成功）
├── 避免KR变成任务列表
└── 检查点：KR是否SMART

步骤5：关联 Initiatives
├── 识别实现KR的关键行动
├── Initiatives是行动，不是结果
├── 不强求每个KR都有Initiatives
└── 检查点：Initiatives是否对齐KR
```

### 阶段三：对齐和沟通（第6-7步）

```
步骤6：上下对齐
├── 与上级/公司OKR对齐
├── 识别依赖关系
├── 消除重复和冲突
└── 检查点：是否已对齐

步骤7：横向对齐
├── 与相关团队对齐
├── 识别协作需求
├── 建立协同机制
└── 检查点：是否已横向对齐
```

### 阶段四：执行和追踪（第8-9步）

```
步骤8：制定执行计划
├── 分解KR到每月/每周
├── 确定check-in节奏
├── 建立数据追踪机制
└── 检查点：计划是否可执行

步骤9：定期check-in
├── 每周/双周check-in
├── 更新KR进展
├── 识别风险和阻碍
├── 必要时调整（审慎进行）
└── 检查点：check-in是否规律
```

### 阶段五：复盘（第10步）

```
步骤10：周期末复盘
├── 评分每个KR（0-1分）
├── 分析达成/未达成原因
├── 总结经验教训
├── 制定下周期改进计划
└── 检查点：复盘是否深刻
```

## 4. 边界情况（Edge Cases）

### 输入为空的情况

| 场景 | 处理方式 |
|------|----------|
| 第一次制定OKR | 从简单开始，先理解理念 |
| 没有数据追踪能力 | 从可衡量的KR开始，逐步建立 |
| 团队不理解OKR | 先进行OKR培训 |
| 多层级同时制定 | 先公司后部门 |

### 特殊情况

| 场景 | 处理方式 |
|------|----------|
| 跨部门OKR | 指定牵头方，各方贡献 |
| 外部依赖强的OKR | 明确依赖关系，降低风险 |
| 探索性目标 | 用探索性OKR，接受不确定性 |
| 紧急任务多 | 区分OKR和日常任务 |

### OKR常见问题

| 问题 | 解决方案 |
|------|----------|
| KR变成任务 | 重新定义KR为结果 |
| O太保守 | 鼓励挑战性目标 |
| O太多 | 聚焦3-5个核心 |
| 只关注KR完成 | O同样重要 |

## 5. 错误处理（Error Handling）

### 常见错误

| 错误类型 | 识别特征 | 处理方式 |
|----------|----------|----------|
| OKR=KPI | KR全是任务指标 | 重定义KR为结果 |
| O不鼓舞人心 | O描述模糊或保守 | 重写O |
| KR不可衡量 | "提升用户体验" | 改为可测量的指标 |
| O太多 | 列出10个O | 聚焦3-5个 |
| 周期内频繁调整 | 随时改OKR | 审慎调整，有正当理由 |

### KR问题诊断

| 问题类型 | 识别 | 修正方式 |
|----------|------|----------|
| 任务型KR | "完成XX功能" | 改为结果型"用户满意度提升X%" |
| 无法追踪 | 无数据支撑 | 建立数据或改为可观察 |
| 缺乏挑战 | 轻松可达100% | 提高目标 |
| 过于复杂 | 复合指标 | 拆分为简单指标 |

## 6. 质量验证清单（Quality Checklist）

### Objective质量
- [ ] O简洁有力，1-2句话
- [ ] O鼓舞人心，有激励性
- [ ] O对齐上级/公司目标
- [ ] 每个周期3-5个O
- [ ] O有一定挑战性

### Key Results质量
- [ ] 每个O有2-4个KR
- [ ] KR是可量化的
- [ ] KR有数据可以追踪
- [ ] KR有挑战性（0.7完成是成功）
- [ ] KR是结果，不是任务

### Initiatives质量
- [ ] Initiatives对齐KR
- [ ] Initiatives是行动
- [ ] Initiatives可控可执行

### 执行质量
- [ ] 有定期check-in
- [ ] KR进展有更新
- [ ] 风险和阻碍有记录
- [ ] 必要时调整OKR

### 复盘质量
- [ ] 每个KR有评分
- [ ] 分析达成/未达成原因
- [ ] 总结经验教训
- [ ] 制定下周期改进

```
### O1：打造用户喜爱的产品体验

**KR1**：NPS（净推荐值）从35提升至45
- 衡量：每月问卷调研
- 现状：35（12月）
- 目标：45

**KR2**：核心功能用户满意度达到90%
- 衡量：功能使用后评分>4分（5分制）
- 现状：78%
- 目标：90%

**KR3**：App Store评分从4.2提升至4.5
- 衡量：iOS/Android双平台评分
- 现状：4.2
- 目标：4.5

- 启动用户访谈计划（每周5个用户）

- 上线反馈快速响应机制（24小时回复）

---

### O2：提升产品交付效率

**KR1**：需求交付周期从14天缩短至7天
- 衡量：需求提出到上线的中位天数
- 现状：14天
- 目标：7天

**KR2**：研发效能提升30%
- 衡量：人天产出/投入
- 现状：基准100%
- 目标：130%

**KR3**：线上bug数量减少50%
- 衡量：P0/P1级bug数
- 现状：每月20个
- 目标：10个

**Initiatives**：
- 引入敏捷开发流程
- 建立技术债务清单
- 加强代码审查

---

### O3：构建数据驱动的产品能力

**KR1**：核心指标看板覆盖率达到100%
- 衡量：DAU、留存、转化等10个核心指标
- 现状：60%
- 目标：100%

**KR2**：数据驱动决策占比达到80%
- 衡量：需求中基于数据提出的比例
- 现状：45%
- 目标：80%

**KR3**：完成3个A/B测试实验
- 衡量：已上线且有结论的实验数
- 现状：1个
- 目标：3个

**Initiatives**：
- 搭建埋点管理平台
- 建立数据定义规范
- 举办数据分享会
```

### 失败案例：某OKR问题

**问题特征**：
- O写成"KPI式"
- KR全是任务
- 全部都能100%完成
- 没有人知道其他人的OKR

**问题诊断**：
- 不理解OKR理念
- KR设计不当
- 目标缺乏挑战
- 缺乏透明

**修正方向**：
- 重写O为鼓舞人心
- 重设计KR为可衡量结果
- 提高目标挑战性
- 公开透明OKR

## 8. FAQ（常见问题）

### Q1：OKR和KPI有什么区别？
**A**：
| 对比 | OKR | KPI |
|------|-----|-----|
| 目的 | 目标管理 | 绩效考核 |
| 目标 | 挑战性 | 现实可达 |
| 评分 | 0-1分，0.7成功 | 通常100%完成 |
| 公开 | 全公司透明 | 通常不公开 |
| 调整 | 周期内审慎调整 | 相对固定 |

### Q2：OKR完成率多少算好？
**A**：
- 0.6-0.7是"成功"
- 1.0完成反而说明目标不够挑战
- 0.3-0.4以下需要分析原因
- 趋势比单周期更重要

### Q3：OKR周期怎么定？
**A**：
- 公司/部门：通常季度
- 个人：可以月度
- 战略目标：半年或年度
- 创业公司：可能月度

### Q4：OKR谁来制定？
**A**：
- 自下而上：个人/团队提出
- 自上而下：公司→部门→个人
- 结合：对齐上级，共同讨论
- 协作：跨部门OKR共同制定

### Q5：KR设计要注意什么？
**A**：
- 必须是可衡量的
- 是结果，不是任务
- 有数据可以追踪
- 有一定挑战性
- 避免太多（2-4个足够）

### Q6：OKR周期内可以调整吗？
**A**：
- 原则上不调整
- 重大变化可审慎调整
- 需要说明理由
- 不要频繁调整

### Q7：OKR和绩效考核挂钩吗？
**A**：
- 建议不直接挂钩
- 挂钩会导致目标保守
- OKR应该鼓励挑战
- 可以作为参考因素
