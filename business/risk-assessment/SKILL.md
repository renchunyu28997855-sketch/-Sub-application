# risk-assessment

## 功能说明
系统性地识别和评估风险，包括技术风险、业务风险、合规风险等，并制定应对策略。

---

## 1. 假设声明（Assumptions）

### 前提条件
- 用户有明确的评估对象（项目/业务/产品）
- 用户对评估范围有基本认知
- 评估结果将用于实际决策
- 用户愿意投入时间收集信息

### 适用范围
- 项目风险管理（新项目/迭代项目）
- 业务风险评估（商业计划/战略决策）
- 产品风险评估（新功能/产品线）
- 运营风险评估（业务运营）
- 投资风险评估（项目/公司）
- 供应链风险评估

### 不适用场景
- 需要专业法律咨询的法律风险
- 需要专业财务咨询的复杂金融风险
- 需要专业安全评估的系统性安全风险
- 已发生的安全事故（应走应急流程）

### 评估假设
- 使用定性评估为主（概率×影响）
- 风险值=概率×影响（高/中/低）
- 风险评估是持续性活动，非一次性

## 3. 详细工作流程（Detailed Workflow）

### 阶段一：评估准备（第1-2步）

```
步骤1：明确评估范围和目标
├── 评估对象是什么
├── 涉及哪些业务/技术领域
├── 评估的边界在哪
├── 干系人是谁
└── 检查点：范围是否清晰

步骤2：收集相关信息
├── 项目文档（计划、需求、设计）
├── 历史数据（以往风险、问题）
├── 行业数据（行业风险基准）
├── 专家意见（经验判断）
└── 检查点：信息是否充分
```

### 阶段二：风险识别（第3-4步）

```
步骤3：风险头脑风暴
├── 技术风险（技术选型、系统稳定性、安全）
├── 业务风险（市场变化、竞争、供应链）
├── 运营风险（流程、人员、执行）
├── 合规风险（法规、政策、行业标准）
├── 财务风险（成本、收入、资金）
└── 检查点：是否覆盖所有类别

步骤4：风险梳理和归类
├── 合并重复风险
├── 剔除超出范围的风险
├── 细化模糊风险
└── 检查点：风险列表是否完整且清晰
```

### 阶段三：风险评估（第5-6步）

```
步骤5：评估风险概率
├── 历史数据法（以往类似事件发生频率）
├── 专家判断法（经验估算）
├── 类比法（参考类似项目）
├── 概率分布法（蒙特卡洛模拟）
└── 检查点：概率评估是否有依据

步骤6：评估风险影响
├── 影响范围（局部/广泛）
├── 影响程度（轻微/中等/严重/灾难）
├── 影响时间（短期/中期/长期）
├── 恢复难度（容易/困难/极难）
└── 检查点：影响评估是否合理
```

### 阶段四：风险分析（第7-8步）

```
步骤7：计算风险值
├── 风险值 = 概率 × 影响
├── 高风险：风险值 ≥ 60
├── 中风险：30 ≤ 风险值 < 60
├── 低风险：风险值 < 30
└── 检查点：风险排序是否合理

步骤8：确定风险优先级
├── TOP风险（需立即处理）
├── 中等风险（需制定应对）
├── 低风险（持续监控）
└── 检查点：优先级是否与实际匹配
```

### 阶段五：应对策略（第9-10步）

```
步骤9：制定应对策略
├── 规避策略（消除风险源）
├── 转移策略（保险、外包）
├── 缓解策略（降低概率或影响）
├── 接受策略（接受残余风险）
└── 检查点：策略是否可行且有效

步骤10：分配责任和资源
├── 风险责任人
├── 应对所需资源
├── 时间安排
└── 检查点：责任是否明确
```

### 阶段六：监控和复盘（第11步）

```
步骤11：建立监控机制
├── 监控指标
├── 检查频率
├── 告警阈值
├── 升级机制
└── 检查点：机制是否可执行

步骤12：定期复盘
├── 风险应对效果评估
├── 新风险识别
├── 风险管理改进
└── 检查点：是否持续优化
```

## 4. 边界情况（Edge Cases）

### 输入为空的情况

| 场景 | 处理方式 |
|------|----------|
| 用户只说"评估一下风险" | 询问评估对象和范围，引导定义评估边界 |
| 没有历史数据 | 使用类比法参考类似项目 |
| 团队经验不足 | 引入外部专家意见 |
| 涉及专业领域 | 建议咨询专业人士（法务、财务、安全） |

### 评估极端情况

| 场景 | 处理方式 |
|------|----------|
| 所有风险都是高风险 | 重新评估，可能是标准过宽 |
| 没有中低风险 | 检查是否识别充分 |
| 风险之间相互关联 | 识别风险关联性，整体评估 |
| 新兴领域无参考 | 专家判断为主，保守估计 |

### 特殊情况

| 场景 | 处理方式 |
|------|----------|
| 法规政策不确定 | 标注为"高度不确定"，准备备选方案 |
| 技术路线不确定 | 考虑多个技术方案的风险 |
| 依赖外部合作方 | 重点评估合作方风险 |
| 资金链紧张 | 财务风险作为TOP风险 |

## 5. 错误处理（Error Handling）

### 常见错误

| 错误类型 | 识别特征 | 处理方式 |
|----------|----------|----------|
| 风险识别不全面 | 只有常见风险，遗漏领域特殊风险 | 引入专家或使用风险清单 |
| 概率高估 | 过度自信，历史数据被忽视 | 引入客观数据 |
| 影响低估 | "没问题"心态，忽视严重后果 | 考虑最坏情况 |
| 风险值失真 | 概率和影响独立评估 | 考虑风险关联性 |
| 应对策略不具体 | "加强注意"等模糊策略 | 制定具体可执行措施 |

### 评估偏差

| 偏差类型 | 说明 | 处理方式 |
|----------|------|----------|
| 幸存者偏差 | 只看成功案例 | 引入失败案例分析 |
| 近期偏差 | 过度重视近期事件 | 拉长时间维度 |
| 群体思维 | 团队讨论时随大流 | 匿名收集意见 |
| 乐观偏差 | 预期过于乐观 | 引入魔鬼辩护人角色 |

## TOP风险清单

### 🔴 风险1：支付系统故障
- **描述**：支付通道故障导致无法下单
- **概率**：低（10%）
- **影响**：极高（直接损失100%订单收入）
- **风险值**：80（高）
- **应对策略**：
  - 主备支付通道自动切换
  - 接入支付宝、微信、云闪付3家以上
  - 建立支付监控告警（响应时间>3秒告警）
  - 制定支付故障应急预案
- **责任人**：支付负责人

### 🔴 风险2：供应链断裂
- **描述**：供应商断货导致爆款缺货
- **概率**：中（30%）
- **影响**：高（影响GMV和用户体验）
- **风险值**：60（高）
- **应对策略**：
  - 每款爆款至少2家供应商
  - 建立安全库存机制（爆款库存>7天销量）
  - 供应商分级管理，核心供应商深度合作
  - 开发供应商备选池
- **责任人**：供应链负责人

---

## 🟡 中等风险

### 风险4：获客成本超支
- **概率**：高（50%）
- **影响**：中
- **应对**：
  - 设定CAC上限红线
  - 建立获客ROI监控日报
  - 多元化获客渠道，降低单一依赖

### 风险5：核心人员离职
- **概率**：中（25%）
- **影响**：高
- **应对**：
  - 关键岗位AB角配置
  - 核心文档不放在个人手中
  - 股权和绩效绑定

---

## 🟢 低风险（持续关注）

- 竞争对手价格战：准备价格保护策略
- 物流时效不达标：选择优质物流合作商，建立KPI考核

## 8. FAQ（常见问题）

### Q1：风险评估多久做一次？
**A**：
- 项目型：项目启动时、重大变更时、里程碑节点
- 业务型：季度评估
- 持续性风险：建立实时监控

### Q2：概率怎么估算？
**A**：
- 有历史数据：用频率统计
- 无历史数据：专家估算
- 参考类似项目：类比法
- 结合多种方法：三角分布

### Q3：影响怎么衡量？
**A**：
- 财务影响：金额损失
- 时间影响：延期多久
- 声誉影响：品牌损失程度
- 客户影响：用户流失数量
- 法规影响：合规风险

### Q4：风险值多少算高？
**A**：
- 高风险：风险值≥60（需立即处理）
- 中风险：30≤风险值<60（需制定应对）
- 低风险：风险值<30（持续监控）
- 具体阈值根据行业和公司标准调整

### Q5：无法消除的风险怎么办？
**A**：
1. 评估是否可接受
2. 如不可接受：转移（保险/外包）或缓解
3. 如可接受：制定应急预案，准备残余风险
4. 持续监控，准备触发条件

### Q6：团队不重视风险怎么办？
**A**：
- 用数据说话（历史案例、量化分析）
- 关联到业务目标（影响可视化）
- 建立奖惩机制（风险管理纳入考核）
- 定期复盘（用事实证明价值）
