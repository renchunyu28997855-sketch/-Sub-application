# project-plan

## 功能说明
帮助制定完整的项目计划，包括时间线、里程碑、资源分配和风险管理。

---

## 1. 假设声明（Assumptions）

### 前提条件
- 用户有明确的项目目标和交付物
- 项目有明确的开始和结束时间
- 项目团队成员和角色已知
- 项目预算或资源限制已知

### 适用范围
- 新产品开发项目
- 产品迭代升级项目
- 技术改造项目
- 市场推广活动
- 运营活动策划
- 跨团队协作项目
- 内部工具开发

### 不适用场景
- 日常运维工作
- 紧急故障处理（应用事件响应流程）
- 纯研究性/探索性工作（无法规划）
- 长期持续性运营工作（应用OKR管理）

### 环境假设
- 项目使用敏捷开发（Scrum/看板）
- 团队规模5-20人
- 项目周期1-6个月
- 需要跨职能团队参与

## 3. 详细工作流程（Detailed Workflow）

### 阶段一：项目理解（第1-3步）

```
步骤1：明确项目背景
├── 项目起源和动机
├── 与公司战略的关系
├── 成功标准是什么
└── 检查点：是否有清晰的目标

步骤2：定义项目范围
├── 包含什么（In Scope）
├── 不包含什么（Out of Scope）
├── 可协商的范围（Negotiable）
└── 检查点：范围是否与目标匹配

步骤3：识别干系人
├── 核心团队（项目成员）
├── 决策者（项目负责人/CTO）
├── 业务方（需求提出方）
├── 资源方（其他部门支持）
└── 检查点：是否识别所有相关方
```

### 阶段二：任务分解（第4-5步）

```
步骤4：WBS任务分解
├── 阶段1
│   ├── 任务1.1
│   ├── 任务1.2
│   └── 任务1.3
├── 阶段2
│   ├── 任务2.1
│   └── 任务2.2
└── 检查点：任务是否分解到可分配级别

步骤5：任务关联分析
├── 任务依赖关系
├── 串行任务 vs 并行任务
├── 关键路径识别
└── 检查点：依赖关系是否正确
```

### 阶段三：计划制定（第6-8步）

```
步骤6：工期估算
├── 估算方法（专家/类比/参数）
├── 考虑因素（复杂度、风险、干扰）
├── 预留buffer（关键路径任务）
└── 检查点：估算是否合理

步骤7：资源分配
├── 人力分配（谁做什么）
├── 设备/工具需求
├── 预算分配
└── 检查点：资源是否足够

步骤8：制定时间表
├── 甘特图制作
├── 里程碑设定
├── 关键节点确定
└── 检查点：时间表是否可行
```

### 阶段四：风险管理（第9-10步）

```
步骤9：风险识别
├── 技术风险
├── 业务风险
├── 资源风险
├── 外部风险
└── 检查点：风险是否全面

步骤10：风险应对
├── 风险优先级排序
├── 应对策略制定
├── 监控机制建立
└── 检查点：应对是否可行
```

### 阶段五：执行准备（第11步）

```
步骤11：项目启动准备
├── 项目启动会（Kickoff）
├── 团队规则制定
├── 沟通机制建立
├── 文档规范确定
└── 检查点：是否具备启动条件
```

## 4. 边界情况（Edge Cases）

### 输入为空的情况

| 场景 | 处理方式 |
|------|----------|
| 用户只说"帮我做项目计划" | 询问：什么项目？目标是什么？团队多大？ |
| 没有明确结束时间 | 询问关键里程碑，反推时间表 |
| 没有团队名单 | 提供组织架构模板，引导用户填充 |
| 预算未知 | 先按常规资源配置，后续调整 |

### 输入格式错误

| 场景 | 处理方式 |
|------|----------|
| 任务周期过长（>1个月） | 拆分为多个阶段 |
| 任务依赖关系混乱 | 重新梳理依赖链 |
| 资源超负荷 | 调整时间表或增加资源 |
| 里程碑过于密集 | 合并或删除次要里程碑 |

### 特殊情况

| 场景 | 处理方式 |
|------|----------|
| 项目范围不确定 | 分期交付，先做MVP |
| 团队跨地域 | 增加沟通buffer，考虑时差 |
| 依赖外部资源 | 标注外部依赖，跟踪进度 |
| 项目周期过长（>6个月） | 分成多个阶段/ sprint |
| 多项目并行 | 标注资源冲突，优先级排序 |

## 5. 错误处理（Error Handling）

### 常见问题

| 问题类型 | 识别特征 | 处理方式 |
|----------|----------|----------|
| 估算过于乐观 | 所有任务刚好卡在deadline | 增加buffer（关键路径15-20%） |
| 范围蔓延 | 需求不断增加 | 明确范围冻结，变更走流程 |
| 资源不足 | 多任务指向同一资源 | 调整时间表或增加资源 |
| 依赖遗漏 | 某任务突然发现缺前置 | 立即补充依赖，重新排期 |
| 里程碑太多 | 里程碑间距<1周 | 合并次要里程碑 |

### 计划调整

| 场景 | 调整方式 |
|------|----------|
| 某任务延期 | 评估对关键路径影响，决定是否赶工 |
| 需求变更 | 评估影响，更新计划，通知干系人 |
| 人员变动 | 重新分配任务，补充交接 |
| 外部依赖延期 | 调整依赖任务时间，或寻找替代方案 |

## 6. 质量验证清单（Quality Checklist）

### 目标清晰性
- [ ] 项目目标SMART（具体/可衡量/可达成/相关/时限）
- [ ] 成功标准明确
- [ ] 干系人期望已识别

### 范围完整性
- [ ] 工作分解结构（WBS）完整
- [ ] 范围边界清晰
- [ ] 不在范围内的事项已标注

### 计划可行性
- [ ] 任务分解到可执行级别（2-5天）
- [ ] 任务依赖关系正确
- [ ] 关键路径已识别
- [ ] 里程碑设置合理
- [ ] 预留buffer充足

### 资源匹配性
- [ ] 每项任务有明确负责人
- [ ] 资源负荷平衡
- [ ] 无资源冲突

### 风险可控性
- [ ] 主要风险已识别
- [ ] 风险等级评估合理
- [ ] 应对策略明确
- [ ] 监控机制建立

### 沟通有效性
- [ ] 沟通机制明确（周会/日会）
- [ ] 汇报模板确定
- [ ] 升级路径清晰

### 失败案例：某项目计划问题

**问题特征**：
- 里程碑只有"完成"
- 所有任务没有依赖关系
- 资源分配表只有人数没有具体人
- 风险清单写着"无"

**问题诊断**：
- 里程碑不具体，无法判断进度
- 任务有依赖但未标注
- 资源分配不具体，无法执行
- 风险识别不充分

**修正方向**：
- 里程碑增加交付物和日期
- 补充任务依赖关系
- 明确每个任务的负责人
- 全面识别风险

## 8. FAQ（常见问题）

### Q1：项目计划需要多详细？
**A**：详细到每项任务2-5天，有明确负责人和交付物即可。不需要分解到小时，太细反而难以管理。

### Q2：敏捷项目还需要详细计划吗？
**A**：敏捷项目有Product Backlog和Sprint计划，详细程度取决于团队需要。但Epic级别的计划和大里程碑仍需提前确定。

### Q3：需求不确定怎么办？
**A**：分两期交付，第一期做MVP确认方向，第二期根据反馈调整。计划中标注"待确认"项，后续补充。

### Q4：资源不足怎么办？
**A**：
1. 评估是否可以延期
2. 评估是否可以砍功能
3. 评估是否可以借调资源
4. 评估是否可以外包
5. 与管理层沟通优先级

### Q5：项目延期了怎么办？
**A**：
1. 分析延期原因
2. 评估对里程碑的影响
3. 决定：赶工/加资源/缩减范围/延期
4. 更新计划，通知干系人
5. 复盘，避免下次同类问题

### Q6：如何让团队遵守计划？
**A**：
- 计划是团队共同制定的
- 定期检查和同步
- 可视化（看板/甘特图）
- 及时调整但不随意变更
- 庆祝里程碑达成
