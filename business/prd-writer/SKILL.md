# prd-writer

## 功能说明
帮助撰写专业的PRD产品需求文档，确保需求清晰、完整、可执行。

---

## 1. 假设声明（Assumptions）

### 前提条件
- 用户有明确的产品背景和目标
- 需求来自真实用户或业务场景
- 产品经理对产品方向有决策权
- 开发团队有技术可行性认知

### 适用范围
- 新功能开发PRD
- 功能优化PRD
- 产品V1.0整体PRD
- 技术改造PRD
- 运营活动PRD
- B端产品需求文档

### 不适用场景
- 极度简化的内部工具（可用更轻量文档）
- 纯技术架构文档（应使用技术设计文档）
- 外部合作协议（应使用合同模板）
- 市场分析报告（应使用市场分析文档）

### 协作假设
- PRD将用于团队评审和开发参考
- 需要技术团队确认可行性
- 需要设计团队确认体验方案
- 需要测试团队确认验收标准

## 2. 输入输出（I/O Format）

### 输入格式

**必需信息**：
```
产品名称：[名称]
功能名称：[名称，如是新功能则必填]
文档版本：[如V1.0]
产品经理：[姓名]
```

**可选补充**：
- 产品背景和目标
- 需求来源（用户反馈/数据/业务）
- 期望上线时间
- 相关需求文档（已有功能/系统）
- 技术约束说明

### 输出格式

```
【产品名称】XXX
【文档版本】V1.0
【更新日期】XXX
【产品经理】XXX
【PRD正文】
...
```

## 3. 详细工作流程（Detailed Workflow）

### 阶段一：需求理解（第1-3步）

```
步骤1：明确产品背景
├── 解决什么问题/满足什么需求
├── 目标用户是谁
├── 业务目标是什么
├── 与公司战略的关系
└── 检查点：背景是否清晰、有依据

步骤2：收集需求信息
├── 原始需求来源（用户反馈/数据/业务方）
├── 竞品参考（如何解决类似问题）
├── 技术约束（技术可行性）
├── 数据埋点需求
└── 检查点：信息是否充分

步骤3：确定需求范围
├── 核心功能（必须做）
├── 扩展功能（可以做）
├── 未来功能（暂不做）
└── 检查点：范围是否清晰、与资源匹配
```

### 阶段二：结构设计（第4-5步）

```
步骤4：设计产品方案
├── 功能架构（模块划分）
├── 用户流程（核心路径）
├── 页面结构（信息架构）
├── 关键交互（特殊状态/异常）
└── 检查点：方案是否完整、可落地

步骤5：确定验收标准
├── 功能验收条件
├── 性能指标要求
├── 兼容性要求
├── 边界条件处理
└── 检查点：标准是否可测试、可衡量
```

### 阶段三：文档撰写（第6-7步）

```
步骤6：撰写PRD正文
├── 产品概述
├── 用户故事/用例
├── 功能需求详述
├── 非功能需求
├── UI/UX要求
├── 验收标准
├── 风险与依赖
└── 检查点：是否覆盖所有要点

步骤7：自审与修改
├── 逻辑是否通顺
├── 是否有遗漏
├── 表述是否清晰无歧义
├── 技术可行性确认
└── 检查点：文档质量是否达标
```

### 阶段四：评审与发布（第8-9步）

```
步骤8：组织评审
├── 评审参与者：产品/设计/技术/测试/运营
├── 评审内容：背景/方案/验收
├── 评审纪要：记录问题和建议
└── 检查点：是否达成共识

步骤9：版本管理
├── 评审后修改版本
├── 需求变更记录
├── 文档归档
└── 检查点：版本是否正确
```

## 4. 边界情况（Edge Cases）

### 输入为空的情况

| 场景 | 处理方式 |
|------|----------|
| 用户只说"帮我写PRD" | 询问：什么产品的什么功能？背景是什么？ |
| 没有需求来源 | 引导梳理需求背景，或建议先做用户调研 |
| 需求模糊不清 | 拆解为更小颗粒度，逐步明确 |
| 多需求混杂 | 拆分为多个独立PRD |

### 输入格式错误

| 场景 | 处理方式 |
|------|----------|
| 功能描述抽象 | 转化为具体功能点和用户场景 |
| 验收标准主观 | 转化为可测试的量化指标 |
| 范围蔓延 | 明确本期范围，标注未来需求 |
| 依赖关系不清 | 梳理并明确前置依赖 |

### 特殊情况

| 场景 | 处理方式 |
|------|----------|
| 技术方案未定 | PRD中标注技术方案待确认 |
| 跨团队依赖 | 单独列出依赖项和负责人 |
| 敏感功能 | 标注特殊权限或合规要求 |
| AB测试需求 | 明确测试目标和分流方案 |
| 数据相关需求 | 明确埋点需求和指标定义 |

## 5. 错误处理（Error Handling）

### 常见问题

| 问题类型 | 识别特征 | 处理方式 |
|----------|----------|----------|
| 需求模糊 | "用户应该可以..."、"大概..." | 改写为具体描述 |
| 验收标准主观 | "体验要好"、"流程要顺" | 转化为可测试指标 |
| 范围蔓延 | 功能点超过10个且关联复杂 | 拆分为多个PRD |
| 依赖遗漏 | 未标注外部依赖 | 补充依赖清单 |
| 异常处理缺失 | 只有主流程描述 | 补充异常场景 |
| 性能要求缺失 | 无性能指标 | 按常规标准补充 |

### 歧义处理

| 歧义类型 | 示例 | 处理方式 |
|----------|------|----------|
| 数量歧义 | "大量用户" | 明确具体数量 |
| 时间歧义 | "尽快"、"及时" | 明确时间标准 |
| 程度歧义 | "优先"、"重要" | 明确判断标准 |
| 范围歧义 | "相关功能" | 列举具体功能 |

## 6. 质量验证清单（Quality Checklist）

### 内容完整性

- [ ] 产品概述包含背景、目标、范围
- [ ] 用户故事覆盖主要用户类型
- [ ] 功能需求描述完整
- [ ] 非功能需求包含性能、安全、兼容性
- [ ] UI/UX要求明确
- [ ] 验收标准具体可测试
- [ ] 风险和依赖已识别
- [ ] 术语表/缩写说明（如有）

### 逻辑清晰性

- [ ] 背景 → 目标 → 方案的逻辑通顺
- [ ] 主流程 → 异常流程覆盖完整
- [ ] 功能描述无歧义
- [ ] 层级结构清晰（模块/子模块/功能点）
- [ ] 术语使用一致

### 可执行性

- [ ] 技术可行性已确认
- [ ] 验收标准可测试
- [ ] 依赖关系已明确
- [ ] 优先级已标注
- [ ] 资源需求已估算

### 评审质量

- [ ] 评审参与者覆盖所有相关方
- [ ] 评审问题已记录
- [ ] 评审决议已达成
- [ ] 修改已同步到文档
- [ ] 评审纪要已归档

## 7. 多个示例（Multiple Examples）

### 典型案例：签到功能PRD

**输入**：
```
产品：App签到功能
功能：每日签到奖励
背景：提升DAU和留存
```

**PRD输出片段**：
```
# 用户签到功能 PRD

## 1. 产品概述

### 1.1 背景
为了提升用户活跃度和留存率，计划上线每日签到功能，通过连续签到奖励机制激励用户形成使用习惯。

### 1.2 目标
- DAU提升10%
- 次日留存率提升5%
- 连续7天签到用户占比达30%

### 1.3 范围
- 用户端：签到入口、签到日历、奖励展示
- 管理端：签到规则配置、数据统计

---

## 2. 用户故事

| 角色 | 故事 |
|------|------|
| 普通用户 | 作为一个用户，我希望每日签到能获得积分奖励，这样我有动力每天打开App |
| 活跃用户 | 作为一个活跃用户，我希望有连续签到奖励，这样我会更积极地每天签到 |
| 沉默用户 | 作为一个沉默用户，我希望有补签机会，这样偶尔忘记签到也不会完全断签 |

---

## 3. 功能需求

### 3.1 签到入口
- 位置：首页底部Tab或悬浮按钮
- 样式：圆形按钮，点击有动效
- 状态：
  - 未签到：显示"签到"按钮
  - 已签到：显示"已签到"+"连续X天"
  - 补签中：显示"补签"按钮

### 3.2 签到奖励规则
| 连续签到天数 | 奖励积分 |
|-------------|---------|
| 1天 | 10分 |
| 2天 | 15分 |
| ... | ... |
| 7天+ | 50分（封顶） |

---

## 4. 非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | 签到接口响应时间<200ms |
| 可用性 | 签到功能可用性≥99.9% |
| 兼容性 | iOS 12+, Android 8+ |
| 数据 | 签到数据保留2年 |

---

## 5. 验收标准

### 5.1 功能验收
- [ ] 用户点击签到按钮后，积分正确增加
- [ ] 连续签到天数正确计算
- [ ] 签到日历正确显示当月签到状态
- [ ] 补签功能正常使用
- [ ] 奖励规则按表格执行

### 5.2 异常情况
- [ ] 网络异常时提示用户重试
- [ ] 重复签到时提示"今日已签到"
- [ ] 补签卡不足时提示获取方式
```

---

### 边界案例：B端数据看板需求

**输入**：
```
产品：运营数据看板
功能：实时数据展示
对象：内部运营人员
特点：数据量大、需要筛选、多人同时使用
```

**PRD要点**：
- 需明确数据源和更新频率
- 需标注权限控制要求
- 需说明导出功能
- 需考虑加载性能

---

### 失败案例：某PRD问题分析

**问题PRD特征**：
- 背景写的是"公司战略"，与功能无直接关联
- 功能描述是"实现一个强大的数据分析功能"
- 验收标准是"让用户满意"
- 没有异常流程

**问题诊断**：
- 背景不够具体，无法判断优先级
- 功能描述无法指导开发
- 验收标准无法测试
- 缺少异常场景

**修正方向**：
- 写具体的问题和用户痛点
- 拆分为具体功能点
- 制定可测试的验收标准
- 补充异常流程

## 8. FAQ（常见问题）

### Q1：PRD应该多详细？
**A**：核心功能详细写（流程、规则、状态），通用功能简略写。原则：开发看了知道怎么做，测试看了知道怎么测。

### Q2：PRD写错了怎么办？
**A**：走需求变更流程：评估影响→通知相关方→更新文档→确认

### Q3：技术方案也要写进PRD吗？
**A**：不需要。PRD是产品视角，技术方案由技术团队输出。但如果技术方案影响产品决策（如技术限制导致方案调整），应在PRD中说明。

### Q4：多个需求一起写还是分开写？
**A**：
- 相关功能：可以合为一个PRD
- 独立功能：建议分开，便于管理
- 大功能+小优化：建议分开

### Q5：PRD评审没通过怎么办？
**A**：
1. 记录评审问题
2. 分类：设计问题/技术问题/需求问题
3. 逐一解决
4. 重新评审或邮件确认

### Q6：如何让开发更配合评审？
**A**：
- 提前沟通，预览PRD
- 评审前发送文档，有时间阅读
- 评审时聚焦问题，不讨论实现细节
- 评审后及时同步结论
