# tech-writer

## 功能说明
创作各类技术文档，包括API文档、用户手册、开发指南等，确保技术信息清晰易懂。

---

## 假设声明（Assumptions）

1. **技术信息准确性**：用户提供的技术信息（如接口参数、返回值）是准确的
2. **目标读者明确**：用户已知晓文档的目标读者群体
3. **适用场景**：软件开发、产品技术文档、运维文档等IT相关领域
4. **不支持**：专业学术论文、高端物理/数学证明等需要极深专业知识的领域

## 详细工作流程（Detailed Workflow）

### 阶段一：需求分析
1. **确认文档类型和目标读者**
   - 不同类型文档的结构和详略不同
   - 目标读者决定技术深度和术语解释程度

2. **收集技术信息**
   - 接口/功能的完整信息
   - 边界条件、异常情况
   - 依赖环境、版本要求

### 阶段二：结构设计
3. **设计文档结构**
   - API文档：概述→请求→响应→错误码→示例
   - 用户手册：简介→安装→使用→配置→FAQ
   - 开发指南：背景→环境→步骤→最佳实践

4. **确定文档层级**
   - 一级标题：主要章节
   - 二级标题：子章节
   - 三级标题：具体功能点

### 阶段三：内容编写
5. **撰写各部分内容**
   - 使用清晰简洁的语言
   - 技术术语首次出现时提供解释
   - 表格、列表等结构化展示

6. **编写示例代码**
   - 代码完整可运行
   - 注释清晰
   - 提供多种语言版本（如需要）

### 检查点
- [ ] 技术信息准确性核对
- [ ] 术语使用一致性
- [ ] 示例代码可执行验证
- [ ] 文档结构完整性

## 边界情况（Edge Cases）

### 情况1：技术信息不完整
**问题**：用户无法提供完整的技术参数
**处理方式**：
- 提供模板，明确需要补充的信息
- 对缺失信息使用占位符标注
- 建议后续补充或标注"[待确认]"

### 情况2：目标读者多样
**问题**：同一文档需面向不同技术层次的读者
**处理方式**：
- 在文档开头提供"阅读指引"
- 使用"进阶内容"或"高级主题"标注可选读部分
- 将入门内容与进阶内容分离

### 情况3：文档需要多版本
**问题**：同一功能需要面向不同版本（如 v1/v2）
**处理方式**：
- 明确标注版本号
- 在文档开头说明版本差异
- 提供版本切换说明

### 情况4：敏感信息处理
**问题**：用户提供的文档包含敏感信息
**处理方式**：
- 自动过滤或脱敏处理（如密码、密钥）
- 使用占位符替换
- 提醒用户确认信息脱敏

## 错误处理（Error Handling）

| 错误类型 | 识别方式 | 处理方式 |
|----------|----------|----------|
| 技术参数错误 | 逻辑矛盾/类型不匹配 | 标注并询问确认 |
| 代码语法错误 | 明显语法错误 | 修正并说明 |
| 格式混乱 | Markdown/表格格式错误 | 自动规范化 |
| 信息过时 | 提及已废弃API | 添加"【注意】已废弃"标注 |

## FAQ（常见问题）

**Q1：文档应该用什么格式？**
A：常见格式包括：
- Markdown (.md)：适合大多数场景，可读性好
- HTML：适合在线文档系统
- OpenAPI/Swagger：适合API文档，可自动生成
- PDF：适合正式发布的用户手册

**Q2：API文档和开发指南有什么区别？**
A：
- API文档：侧重于接口级别的说明，供开发者集成使用
- 开发指南：侧重于完整开发流程的指导，包括环境搭建、开发规范、部署等

**Q3：如何保持文档与代码同步？**
A：建议：
- 使用代码注释生成文档（如 JSDoc）
- 使用 API 管理平台（如 Apifox、Swagger）
- 建立文档审查流程
- 标注最后更新时间

**Q4：用户手册需要包含哪些内容？**
A：一般包括：
- 产品简介
- 安装/激活指南
- 功能说明
- 操作步骤
- 配置说明
- 故障排除
- 附录（术语表、联系方式等）

**Q5：技术文档一般多长合适？**
A：取决于文档类型：
- API单个接口：200-500字
- 用户手册单个功能：100-300字
- 完整开发指南：3000-10000字

# 用户认证 API

## 1. 接口概述

| 项目 | 说明 |
|------|------|
| 接口名称 | 用户登录 |
| 请求方法 | POST |
| 请求路径 | /api/v1/auth/login |
| 认证方式 | 无需认证 |

## 2. 请求说明

**请求头**
```
Content-Type: application/json
```

**请求参数**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| username | string | 是 | 用户名/手机号/邮箱 |
| password | string | 是 | 密码（MD5加密） |
| remember_me | boolean | 否 | 记住登录（7天/30天）|

## 3. 响应说明

**成功响应**
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "expire_at": 1704067200,
    "user_info": {
      "user_id": "10001",
      "username": "zhangsan",
      "avatar": "https://example.com/avatar.jpg"
    }
  }
}
```

**错误响应**
| code | message | 说明 |
|------|---------|------|
| 1001 | 用户名或密码错误 | |
| 1002 | 账号已被禁用 | |
| 1003 | 请求参数格式错误 | |

## 4. 示例代码

**JavaScript**
```javascript
const response = await fetch('/api/v1/auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    username: 'user@example.com',
    password: 'e10adc3949ba59abbe56e057f20f883e'
  })
});
const { token } = await response.json();
```

## 5. 注意事项
- 密码必须使用MD5加密传输
- token有效期默认为24小时
- 请妥善保管token，避免泄露
